# Design: AutoGuide Platform

Epic design for the full AutoGuide Documentation Intelligence Engine.
Generated by `@feature-intake` from [docs/PRD.md](../../docs/PRD.md) and grill-me decisions.

## Problem & Intent

Developers building process-heavy apps (HR, operations, admin) ship faster than they document. AutoGuide binds documentation to the application itself — scanning code, runtime DOM, and Playwright traces to produce verifiable knowledge with provenance and confidence. The primary user (developer) installs, scans, reviews, and publishes documentation so employees need less manual training.

**Intent:** Deliver the full Combined Spec as architecture end-state, implemented in five vertical delivery layers with React as reference implementation and browo-hr as primary dogfood target.

## Non-Goals

- Hosted SaaS dependency for core features
- Replacing Playwright as test framework
- Modifying host app business logic without developer action
- Cloud AI by default for HR/person data
- Building all adapters before React reference cycle is proven

## Assumptions

- Solo developer + Cursor agents implement slices via `@ecc-runner`
- browo-hr (`../browo-hr`) has Playwright e2e tests importable for flow discovery
- pnpm monorepo, TypeScript strict, Vitest for unit tests
- JSON is source of truth; SQLite is index/cache only
- German UI copy; English code/commits

## Options Considered

### Option A: MVP-only first (rejected)
- Summary: Ship Help Widget + DOM scan only, defer Playwright/AI/adapters
- Pros: Faster first release
- Cons: Wow moment (step-by-step flows) not achievable; throws away spec architecture
- Evidence: Grill-me decision for full spec (c)

### Option B: Big-bang monolith (rejected)
- Summary: Single issue builds entire platform
- Pros: No integration seams
- Cons: Unreviewable, untestable, blocks ecc-runner pipeline
- Evidence: slice-rules.md max ~800 LOC per issue

### Option C: Layered vertical slices with interface-first core (chosen)
- Summary: Core interfaces → React reference → scanners → confidence → Playwright/AI → UI/publish → other adapters
- Pros: Each slice PR-sized; adapters reuse proven APIs; browo-hr validates early
- Cons: Longer total calendar time
- Evidence: SPEC_FULL.md phases 1–8 map cleanly to layers

### Option D: YAGNI / defer everything except DOM inventory (Ponytail Rung 1)
- Summary: Only monorepo + DOM button list + Markdown dump
- Pros: Smallest possible start
- Cons: Does not meet wow moment or spec acceptance
- **Documented as Rung 1 alternative — not chosen for this epic**

## Decision

**Chosen:** Option C — layered vertical slices, full spec end-state, React + browo-hr first.

**Why:** User locked scope (c), wow moment requires flows not inventory, and plugin architecture must be real from day one to avoid rework for Vue/Angular/Svelte/Tauri/Flutter.

## Implementation Sketch

```
packages/core/          — Fact, Graph, Confidence, Verification, PluginRegistry
packages/config/        — autoguide.config.ts schema
packages/storage/       — JSON writer, SQLite index
packages/cli/           — init, scan, review, generate, export, doctor, publish
plugins/react/          — Provider, Widget, Inspector, DocElement
plugins/vue|angular|svelte/ — same runtime API as React
plugins/tauri|flutter/  — non-DOM runtime bridges
examples/react-vite/    — minimal reference host
```

**Dogfood path:** `examples/react-vite` → `../browo-hr/browo-hr/frontend` (3 flows documented)
