## Was ist dieses Projekt?

AutoGuide ist eine **Documentation Intelligence Engine**: SDK, Scanner, CLI und Runtime-UI, die aus Code, Runtime-Verhalten und Playwright-Traces verifizierbare App-Dokumentation erzeugt — mit Provenance, Confidence und Developer-Review.

**PRD:** [docs/PRD.md](docs/PRD.md)  
**Full Spec:** [docs/SPEC_FULL.md](docs/SPEC_FULL.md)

### Was dieses Repo **ist**

- pnpm Monorepo mit framework-unabhängigem `@autoguide/core`
- CLI (`autoguide init|scan|review|generate|export|doctor`)
- Framework-Adapter (React zuerst, dann Vue/Angular/Svelte/Tauri/Flutter)
- Lokale Wissensbasis in `.autoguide/` (JSON SoT, SQLite Index)
- Help Widget + Inspector für kontextuelle Hilfe

### Was dieses Repo **nicht** ist

- Kein Wiki, kein generischer Chatbot, kein Test-Framework-Ersatz
- Kein gehostetes SaaS für Kernfunktionen
- Kein Tool, das Host-App-Business-Logik ohne Dev-Aktion ändert

## Tech Stack (verbindlich)

| Bereich | Technologie | Notiz |
|---------|-------------|-------|
| Monorepo | pnpm workspaces | packages/, plugins/, examples/ |
| Sprache | TypeScript strict | Kein `any` ohne Begründung |
| Core | `@autoguide/core` | Kein React/DOM/Node ohne Abstraktion |
| React Adapter | `@autoguide/react` | Vite-first, kein Next.js-Zwang |
| Scanner | Playwright | Import bestehender Tests, eigener Crawl als Fallback |
| AI | Ollama default | Cloud: OpenAI-compatible, User API-Key + URL |
| Storage | JSON + SQLite | JSON = SoT, SQLite = Index/Cache |
| Tests | Vitest | Playwright für E2E (verify-ui) |
| UI (Widget) | React + Tailwind | Siehe UI_STYLEGUIDE |

**Nicht verwenden:** Supabase als AutoGuide-Backend, mandatory Cloud-AI, Framework-Imports in Core

## Architektur

```
autoguide/
├── packages/
│   ├── core/           # Types, graph, confidence, verification — NO framework imports
│   ├── config/         # autoguide.config.ts schema + loader
│   ├── storage/        # JSON writer, SQLite index
│   └── cli/            # CLI commands
├── plugins/
│   └── react/          # AutoGuideProvider, Widget, Inspector, DocElement
├── examples/
│   └── react-vite/     # Reference example app
├── integrations/       # Self-contained scan scenarios (fixtures, CI)
└── docs/
```

### Schichtenregeln

1. `@autoguide/core` importiert **niemals** React, Vue, DOM APIs oder Node-spezifische Module ohne Abstraktion
2. Framework-Logik lebt nur in `plugins/*`
3. AI-Provider-Logik bleibt hinter `AIProvider`-Interface — kein Leak in Core
4. Manual review outranks AI; `ai_proposal` darf verified facts nie überschreiben
5. `development` mode zeigt alles dem Dev; `published` nur approved + confidence ≥ 0.85

### Critical boundary

```text
Host App → Framework Adapter → Runtime SDK
CLI → Source/Runtime/Playwright Scanner → Core (Graph, Confidence, Verification) → Storage
```

## Validation

- **Verify (deterministic):** `pnpm run verify` (typecheck + test)
- **Quality gate (ship):** `@ecc-check` — verify + review-ticket + AgentShield (+ verify-ui if UI changed)
- **Generate (derived artifacts):** `autoguide generate tours|recommendations|bundle` — uses existing `.autoguide/` scan output, no re-scan
- **Plugins:** `autoguide.config.json` → `plugins` array (builtin ids or module paths); lifecycle `setup → scan → transform → cleanup` with isolated failures
- **Recommendations:** `recommendations.json` on scan; `autoguide doctor` lists prioritized hints with review-queue links
- **API docs:** `docs/api/` — regenerate with `pnpm run docs:api`
- **Typecheck:** `pnpm typecheck`
- **Test:** `pnpm test`
- **Dev (example):** `pnpm dev` → http://localhost:5173
- **E2E:** `pnpm run test:e2e` (Playwright via @verify-ui when ready)

Run `@ecc-check` before commit/PR. Do not bypass hooks. **Kein shimwrappercheck** in diesem Repo.

## QA Pipeline

```
@feature-intake  →  @ecc-runner  →  @implement  →  @ecc-check  →  @verify-ui
```

- Epic design: `.qa/design/`
- Issue drafts: `.qa/intake/`
- Acceptance: `.qa/acceptance/` (auto-generated by @implement)
- Project config: `.qa/project.yaml`
- Runner profile: `.qa/runner-profile.yaml`

### Ponytail (lazy senior dev)

Reference: [DietrichGebert/ponytail](https://github.com/DietrichGebert/ponytail) (MIT).

| Phase | Role |
|-------|------|
| `@feature-intake` | Rung 1 (YAGNI): dokumentiere was **nicht** Issue wird |
| `@implement` | Kleinster sauberer Diff; 6-rung ladder |
| Architektur | Interfaces vor Adaptern; React-Referenz vor Vue/Angular/Svelte |

**Project rules override Ponytail.** Core-Boundaries, Security, Tests und Accessibility werden nie übersprungen.

## ECC Check (Quality Gate)

Vor Commit/PR: **`@ecc-check`** (nicht shimwrappercheck).

| Phase | Tool |
|-------|------|
| A — Deterministic | `pnpm run verify` |
| B — Acceptance | `@verify-ticket` (wenn `.qa/acceptance/*.md` existiert) |
| C — Code review | `@review-ticket` bis ACCEPT |
| D — Security | `npx ecc-agentshield scan --path .cursor` (siehe `.cursor/README.md`) |
| E — UI | `@verify-ui` wenn UI-Dateien im Diff |
| Ship | `@commit-pr-safe` nur nach **READY** |

Konfiguration: `.qa/project.yaml` → `checksCommand: pnpm run verify`
